Wie ein LLM wirklich funktioniert

Du fängst ja grad mit PyTorch an und baust dein erstes Zeug zusammen. Dann solltest du auch einmal sehen was da eigentlich passiert wenn du einem Sprachmodell einen Satz gibst. Nicht der Marketing-Bullshit, nicht "die KI versteht dich", sondern was wirklich abläuft. Schritt für Schritt.

Wir nehmen einen Satz und jagen ihn einmal komplett durch so ein Modell. Sagen wir du schreibst folgenden Satz:

„Python ist eine schöne Sprache."

Fünf Stationen. An den passenden Stellen bau ich dir PyTorch-Snippets ein damit du siehst wo welches Teil in echtem Code lebt.

Station 1: Der Tokenizer

Das Modell kennt keine Sprache. Keine Wörter, keine Buchstaben, gar nichts. Es kennt nur Zahlen. Riesige Tabellen voller Zahlen.

Damit dein Satz da überhaupt rein kann muss er erstmal in Zahlen übersetzt werden. Das macht der Tokenizer, ein kleines Programm das vor dem eigentlichen Modell läuft. Sein Job istText in Stücke hacken und jedem Stück eine Nummer geben. 

Python   → 15490
ist      →  6127
eine     →  5180
sch      →  1364
öne      → 27094
Spr      → 30689
ache     →  4023
.        →    13

Acht Token. Acht Zahlen. Das wars. Mehr kommt beim Modell nicht an. „Python" ist ein Stück weil das Wort im Trainingstext ständig vorkam. „schöne" wurde in zwei Teile gehackt, „Sprache" auch.

Kannst du direkt ausprobieren:

from transformers import AutoTokenizer
tok = AutoTokenizer.from_pretrained("gpt2")
print(tok.encode("Python ist eine schöne Sprache."))

Die genauen IDs hängen vom Tokenizer ab aber die Struktur ist immer die gleiche.

Warum nicht einfach ganze Wörter?

Klingt erstmal logisch? Dachte ich zuerst auch. Jedes Wort ein Token, fertig. Das Problem ist, die Anzahl möglicher Wörter ist quasi unendlich. Allein im Deutschen, Eigennamen, Fachbegriffe, irre zusammengesetzte Substantive. Donaudampfschifffahrtsgesellschaftskapitän. Da bräuchtest du Millionen Einträge.

Ein anderes Extrem ist jeden Buchstaben einzeln als Token zu nehmen. Geht, aber dann braucht das Modell ewig um überhaupt zu checken was ein Wort bedeutet. „Python" wäre dann sechs Token statt einem.

Der Tokenizer macht den Mittelweg. Häufige Buchstabenfolgen werden zusammengeklebt, seltene bleiben klein. Du weißt ja noch was eine HashMap ist, das funktioniert nach einem ähnlichem Prinzip.

Wie lernt der das?

Bevor das Modell trainiert wird, wird erstmal der Tokenizer trainiert. Auf einem riesigen Haufen Text.

Das Verfahren heißt Byte Pair Encoding, BPE. Ist aber überraschend simpel. Du fängst mit einzelnen Buchstaben an. Dann guckst du welche zwei Buchstaben am häufigsten nebeneinander stehen. Die verschmilzt du zu einem Stück. Das wiederholst du tausende Male.

So entstehen Stücke wie „ist", „der", „ee", „ung". Was im Text oft vorkommt wird ein Stück. Was selten ist bleibt zerteilt.

Und jetzt der Punkt. Wenn der Tokenizer hauptsächlich mit englischen Texten trainiert wurde, lernt er halt englische Häufigkeiten. „the" wird ein Token. „Python" wird ein Token weil Programmiersprachen-Kram überall auf Englisch ist. Aber deutsche Wörter wie „schöne"? Zerteilt. „Sprache"? Zerteilt.

Das ist keine echte Intelligenz. Das ist Zählen.

Warum das nicht egal ist

Unser Satz braucht auf Deutsch acht Token. Auf Englisch wäre „Python is a beautiful language." vielleicht sechs. Russisch noch mehr.

Wer auf Englisch promptet kriegt mehr Text ins Kontextfenster rein. Wer über die API arbeitet zahlt pro Token, also ist derselbe Inhalt auf Russisch einfach teurer. Neuere Tokenizer wie der von GPT-4o machen es besser, aber Englisch bleibt am günstigsten.

Das alles weil irgendwann mal jemand Buchstaben gezählt hat in einer Textsammlung wo Englisch dominant war. So simpel ist das. Aber auch hier muss man sagen gibt es schon modernere Ansätze. Nennt sich Multilingual Tokenizer. Die Dinger werden effizient auf die populärsten Sprachen trainiert. Mixture-of-Experts-artige Tokenisierung gibt es btw. auch, oder sogar reine Byte-Level Tokenizer, denen ist Sprache dann komplett egal.

Wo wir stehen

Aus „Python ist eine schöne Sprache." ist geworden:

15490, 6127, 5180, 1364, 27094, 30689, 4023, 13

Acht Zahlen. Keine Bedeutung, kein Zusammenhang, nicht mal die Info dass „ sch" und „öne" zusammen mal ein Wort waren.

Station 2: Embedding und Position

Jetzt kriegen diese Zahlen einen Ort. Ort haben ist die Vorstufe von Bedeutung haben.

Was passiert

Stell dir eine riesige Tabelle vor. So viele Zeilen wie der Tokenizer Einträge hat, also so 100.000 bis 200.000. Und jede Zeile hat 4.096 Spalten.

Jede Zeile gehört zu einem Token. Unser „Python" mit der Nummer 15490 hat eine eigene Zeile. „ ist" mit 6127 hat eine eigene. In jeder Zeile stehen 4.096 Zahlen.

So eine Reihe Zahlen heißt Vektor. Du hattest Vektoren in Mathe, da warens meistens zwei oder drei Zahlen, also ein Punkt auf einer Fläche oder im Raum. Hier ist es das gleiche Prinzip, nur mit 4.096 Zahlen statt drei. Kann sich kein Mensch mehr vorstellen, aber rechnen geht damit genauso.

Im Kontext von Sprachmodellen sagt man zu so einem Vektor auch Embedding.

In PyTorch ist das eine nn.Embedding. Im Grunde eine Lookup-Tabelle, nichts weiter.

import torch
import torch.nn as nn

emb = nn.Embedding(num_embeddings=150_000, embedding_dim=4096)
ids = torch.tensor([15490, 6127, 5180, 1364, 27094, 30689, 4023, 13])
vectors = emb(ids)
print(vectors.shape)   # torch.Size([8, 4096])

Aus acht Zahlen wird ein Tensor mit Shape [8, 4096]. Tensor ist in PyTorch einfach das Wort für so ein mehrdimensionales Zahlengitter. Hier: acht Vektoren, jeder 4.096 Werte lang.

Warum ist das ein Ort?

Denk an Google Maps. Jeder Punkt hat Koordinaten, Länge und Breite. Zwei Cafés nebeneinander haben fast die gleichen Koordinaten. Eins in Worpswede und eins in Tokio haben komplett andere.

Jetzt stell dir vor die Karte hat nicht zwei sondern 4.096 Dimensionen. Dimension ist einfach eine Koordinatenachse. Zwei Dimensionen ist eine Fläche. Drei ist der Raum um dich rum. Vier kannst du dir schon nicht mehr vorstellen. 4.096 schon gar nicht. Aber dem Computer ist das egal, für den ist jede Dimension einfach eine weitere Zahl.

Und in diesem Raum sitzt jedes Token an einer bestimmten Stelle. „Python" sitzt in der Nähe von „JavaScript", „Code", „Programmierung". Weit weg von „Bratwurst" oder „Mondfinsternis".

Diese Positionen hat niemand hinprogrammiert. Das Modell hat sie beim Training gelernt. Milliarden Sätze gesehen, gemerkt welche Wörter in ähnlichen Kontexten auftauchen, und Wörter die ähnliche Nachbarn haben nah beieinander platziert.

Das ist übrigens der berühmte Befund von Tomáš Mikolov 2013 mit Word2Vec. Nimm den Vektor für König, zieh den Vektor für Mann ab, addier den Vektor für Frau, und du landest in der Nähe von Königin. Vektoren addieren und subtrahieren geht nämlich genau wie mit normalen Zahlen, einfach Stelle für Stelle. Also (3, 5) minus (1, 2) ist (2, 3), checkst du? Nur hier halt mit 4.096 Stellen. Reine Statistik, und am Ende sieht es aus wie Bedeutung. Ziemlich wild eigentlich.

Kleiner Haken

Das Embedding ist noch grob. „Bank" hat hier immer die gleiche Position, egal ob Parkbank oder Sparkasse. Beides startet identisch. Dass das Modell den Unterschied später kapiert, passiert erst in der nächsten Station.

Reihenfolge

Noch ein Problem. Die acht Vektoren sagen nichts darüber wo sie im Satz stehen. Aber Reihenfolge ist wichtig. „Hund beißt Mann" und „Mann beißt Hund" sind nicht dasselbe. Wenn das Modell nur die Wörter sieht aber nicht die Reihenfolge, kann es die beiden Sätze nicht unterscheiden. Bullshit.

Also wird zu jedem Vektor noch eine Positions-Info dazugepackt. Sie sagt dem Modell ob das Token an Stelle eins steht oder an Stelle acht.

Wie das genau gemacht wird hat sich über die Jahre geändert. Das originale Transformer-Paper von 2017 hat dafür Sinus-Wellen genommen, also Schwingungsmuster mit verschiedenen Frequenzen, quasi ein Fingerabdruck pro Position. Heute nutzen die meisten Modelle was namens RoPE, Rotary Position Embedding, das die Vektoren je nach Position leicht dreht. Wen juckt das Detail, wichtig ist: nach diesem Schritt weiß jeder Vektor nicht nur welches Token er ist, sondern auch wo er steht.

Wo wir stehen

Acht Vektoren, jeder 4.096 Werte. „Python" ist nicht mehr nur die Zahl 15490 sondern ein Punkt im Raum, nah an anderen Programmiersprachen-Wörtern, und er weiß dass er ganz vorne im Satz steht.

Aber die acht Vektoren wissen noch nichts voneinander. Jeder guckt nur für sich.

Station 3: Attention

Jetzt wirds spannend. Hier passiert das was den Transformer zum Transformer macht.

Warum müssen die überhaupt miteinander reden?

Schau dir den Satz an. „Python ist eine schöne Sprache."

„schöne" beschreibt nicht abstrakt irgendetwas. Es bezieht sich auf die Sprache, und zwar auf Python. „Sprache" meint hier Programmiersprache, nicht Deutsch oder Französisch. Aber woher soll das Modell das wissen wenn jeder Vektor nur für sich alleine dasitzt?

Die einzelnen Vektoren reichen dafür nicht. Die müssen sich gegenseitig angucken und ihre Infos vermischen können.

Das ist Attention.

Die Grundidee

Für jeden der acht Vektoren wird gefragt: "welche der anderen sieben sind gerade wichtig für mich?". Und dann wird der Vektor so umgebaut dass er die Info aus den wichtigen Nachbarn aufnimmt.

Das wars schon als Idee. Das Komplizierte ist nur wie das Modell entscheidet wer wichtig ist.

Q, K und V

Jeder der acht Vektoren wird in drei Versionen von sich selbst zerlegt. Drei verschiedene Transformationen, alle gelernt. Die heißen Q, K und V. Query, Key, Value.

Q ist die Version mit der ein Vektor andere prüft. K ist die Version an der ein Vektor von anderen geprüft wird. V ist die Version die tatsächlich weitergegeben wird wenns einen Match gibt.

In PyTorch:

W_q = nn.Linear(4096, 4096)
W_k = nn.Linear(4096, 4096)
W_v = nn.Linear(4096, 4096)

Q = W_q(vectors)   # [8, 4096]
K = W_k(vectors)   # [8, 4096]
V = W_v(vectors)   # [8, 4096]

Kurz zu nn.Linear weil das überall im Modell auftaucht. Du kennst lineare Funktionen aus der Schule, y = ax + b, also eine Gerade. nn.Linear ist dasselbe nur statt einer Zahl geht ein ganzer Vektor rein und ein ganzer Vektor kommt raus. Statt der einen Zahl a hast du eine Matrix, also ein ganzes Gitter aus Zahlen, mit dem multipliziert wird. Plus ein Vektor b der dazu addiert wird. Selbes Prinzip wie y = ax + b, nur in vielen Dimensionen auf einmal.

Die Namen Q, K, V klingen nach Frage, Antwort und Inhalt. Ist auch die übliche Erklärung und nicht komplett falsch. Aber eigentlich kommen Q und K aus der gleichen Quelle, nämlich dem Token-Vektor, nur mit verschiedenen gelernten Matrizen multipliziert. Es gibt keinen echten „Fragenden" und keinen „Antwortenden". Es sind zwei verschiedene Sichten auf denselben Vektor und ihr Vergleich ergibt eine Zahl die sagt wie ähnlich sich zwei Tokens sind.

Konkret: was passiert bei „schöne"

„schöne" hat einen Q-Vektor. Der wird mit den K-Vektoren von allen acht Tokens verglichen. Aus jedem Vergleich kommt eine Zahl raus, ein Ähnlichkeits-Score. Acht Scores insgesamt.

Wie vergleicht man zwei Vektoren? Ziemlich simpel eigentlich. Du multiplizierst sie Stelle für Stelle und addierst alles zu einer einzigen Zahl auf. Das heißt Skalarprodukt. Wenn zwei Vektoren in eine ähnliche Richtung zeigen wird die Zahl groß. Wenn nicht, klein oder sogar negativ.

Diese acht Scores werden dann in Wahrscheinlichkeiten umgerechnet. Dafür gibt es Softmax. Was Softmax macht: es nimmt eine Liste von Zahlen, egal wie groß oder klein, und macht daraus Werte zwischen null und eins die zusammen genau Eins ergeben. Also eine Wahrscheinlichkeitsverteilung. Was vorher schon groß war wird relativ groß, was klein war wird sehr klein.

Im Idealfall kommt für „Python" und „Sprache" was hohes raus, sagen wir 0,4 und 0,3. Für „ist" und „eine" auch noch was. Für den Punkt quasi null.

Jetzt wird ein neuer „schöne"-Vektor gebaut. Der ist die gewichtete Summe der V-Vektoren von allen acht Tokens. „Python" hat Gewicht 0,4 gekriegt, also fließen 40 Prozent von Pythons V-Vektor da rein.

In PyTorch:

import torch.nn.functional as F

scores  = Q @ K.transpose(-2, -1) / (4096 ** 0.5)   # [8, 8]
weights = F.softmax(scores, dim=-1)                 # [8, 8]
output  = weights @ V                               # [8, 4096]

Das @ ist Matrixmultiplikation. Die macht im Prinzip das Skalarprodukt von oben für alle Paare auf einmal. Die Division durch die Wurzel von 4.096 ist nur dafür da dass die Scores nicht zu groß werden und Softmax nicht ausrastet.

Ergebnis: „schöne" ist nicht mehr das generische „schöne" aus dem Embedding. Es ist jetzt ein „schöne" das weiß dass es um Python geht und dass Sprache hier Programmiersprache meint.

Das passiert parallel für alle acht Tokens. Jeder kriegt sein eigenes Bild davon wer für ihn wichtig ist.

Noch mehr dazu

Die ganze Berechnung läuft nicht einmal sondern parallel mehrfach. Jeder Durchlauf heißt Attention Head. Verschiedene Heads lernen verschiedene Arten von Beziehungen, einer vielleicht Grammatik, einer Bedeutung, einer Positionen. Am Ende werden alle Ergebnisse zusammengeführt. Multi-Head Attention.

Und das läuft nicht nur einmal pro Satz. Das Modell hat dutzende Layer übereinander und in jedem gibt es neue Attention. „schöne" wird also nicht einmal angereichert sondern viele Male hintereinander, immer feiner.

Die Beispielgewichte oben sind übrigens ausgedacht. Ich illustrier ein Prinzip, keine echte Messung.

Wo wir stehen

Aus acht isolierten Vektoren sind acht vernetzte Vektoren geworden. „schöne" weiß dass es um Python geht. „Sprache" weiß dass Programmiersprache gemeint ist. Das ist das Herzstück des Transformers.

Station 4: Nach der Attention

Attention war sowas wie ein Meeting. Alle acht Tokens haben sich angeguckt, jeder hat mitgenommen was für ihn relevant war.

Aber Infos aufnehmen und Infos verarbeiten sind zwei verschiedene Sachen. Kennst du, du kommst aus nem Gespräch und checkst erst Stunden später was du eigentlich davon halten sollst.

Den Vektoren gehts genauso. Die haben miteinander geredet aber keiner hat das Gehörte verarbeitet. Das macht die zweite Hälfte des Transformer-Blocks. Heißt Feed-Forward Network, FFN, manchmal auch MLP.

Was das FFN macht

Klassisches kleines neuronales Netz. Jeder Vektor wird einzeln durchgeschickt. Kein Austausch zwischen den Tokens hier, jeder für sich.

Der Vektor hat 4.096 Werte. Das FFN bläst ihn erstmal auf, typischerweise aufs Vierfache. Also 16.384 Werte. In diesem breiteren Raum wird eine nicht-lineare Funktion angewendet. Danach wird er wieder auf 4.096 zurückgequetscht.

ffn = nn.Sequential(
    nn.Linear(4096, 16384),
    nn.GELU(),
    nn.Linear(16384, 4096),
)

Nicht-linear, kurz erklärt. Lineare Funktionen sind Geraden, y = ax + b. Nicht-linear ist alles was krumm ist, Kurven, Knicke, Schwellen. Warum das wichtig ist: wenn du lineare Funktionen hintereinander hängst kommt am Ende immer nur wieder eine einzige lineare Funktion raus. Das heißt hundert lineare Schichten können nicht mehr als eine einzige. Wär ja Quatsch. Erst wenn du was Krummes dazwischenpackst wird der Stapel mächtiger als jede einzelne Schicht. Das GELU da oben ist so eine krumme Funktion. Du siehst auch ReLU oder SiLU in anderen Modellen, alles Varianten vom gleichen Prinzip.

Was passiert da inhaltlich? Ehrliche Antwort: niemand weiß es genau. Aber Forschung der letzten Jahre zeigt dass im FFN viel von dem steckt was man „das Wissen" eines Modells nennen würde. Dass Paris die Hauptstadt von Frankreich ist lebt wahrscheinlich nicht in der Attention sondern in den FFN-Schichten. Die funktionieren wie ein riesiges gelerntes Nachschlagewerk, das auf jedes Token einzeln angewendet wird.

Wenn Attention das Mischen ist, ist FFN das Nachdenken.

Residual Connections

Wichtiges Detail. Der neue Vektor der aus Attention oder FFN rauskommt ersetzt nicht den alten. Er wird dazu addiert.

x = x + attention(x)
x = x + ffn(x)

Stell dir vor ein Lektor streicht nichts in deinem Text sondern schreibt nur Notizen an den Rand. Das Original bleibt, die neuen Erkenntnisse kommen dazu.

Warum? Weil tiefe neuronale Netze ohne diesen Trick schwer zu trainieren sind. Wenn jeder Layer den Input komplett ersetzt vergessen die hinteren Layer was am Anfang da war. Mit Residual Connections gibt es einen durchgehenden Pfad wo die ursprüngliche Info immer mitläuft. Jede Schicht muss nur kleine Korrekturen anbringen statt alles neu zu erfinden.

Der Trick kommt aus einem Paper von 2015 über ResNets, ursprünglich für Bildverarbeitung. Heute Standard in allem was tief ist. Wirst du in PyTorch überall sehen.

LayerNorm

Noch so ein Ding. Vor oder nach Attention und FFN sitzt eine Operation namens Layer Normalization. Die macht was Schlichtes: sie sorgt dafür dass die Zahlen im Vektor nicht aus dem Ruder laufen.

Was heißt normalisieren? Du rechnest den Durchschnitt aller Werte im Vektor aus, guckst wie weit sie typischerweise davon abweichen, und schiebst und skalierst alles so dass Durchschnitt ungefähr null ist und die Streuung ungefähr eins. Das wars.

Wenn du durch viele Layer rechnest neigen Werte dazu entweder riesig zu werden oder gegen null zu schrumpfen. Beides ist schlecht fürs Training. LayerNorm kalibriert immer wieder neu. Nicht spannend aber ohne das würde am Ende nichts mehr funktionieren.

Und jetzt 80 mal von vorne

Alles was ich gerade beschrieben hab ist EIN Transformer-Block. Attention plus FFN plus Residuals plus LayerNorms.

In einem echten LLM wird das nicht einmal gemacht sondern dutzende bis hunderte Male hintereinander. GPT-2 hatte 12 solche Blöcke. GPT-3 hatte 96. Aktuelle große Modelle haben so 80 bis über 120.

Jeder Block nimmt die acht Vektoren vom vorherigen Block und schickt acht neue weiter. Gleiche Form, anderer Inhalt. Mit jedem Schritt feiner.

In den unteren Blöcken passiert eher syntaktisches Zeug. Welches Token gehört zu welchem, welche Endung passt wohin. Je weiter oben desto abstrakter. Da gehts um Bedeutung, Weltwissen, das was wir verstehen nennen würden.

Wo wir stehen

Nach 80 Blöcken haben wir acht extrem dichte Vektoren. Gleiche Form wie nach dem Embedding, acht mal 4.096, aber der Inhalt ist von oberflächlich zu tief gewandert.

Jetzt muss daraus ein Wort werden.

Station 5: Wie ein Wort rauskommt

Wir haben acht Vektoren die durch 80 Transformer-Blöcke gelaufen sind. Was da drin steht können wir nicht lesen. Aber das Modell muss jetzt was Konkretes draus machen. Ein Token. Das nächste Token nach dem Punkt.

Vom Vektor zur Wortliste

Der letzte der acht Vektoren steht für den gesamten bisherigen Text. Aus dem wird die Vorhersage gebaut.

Dafür gibts den LM-Head, Language Model Head. Konzeptuell das Gegenteil vom Embedding.

Beim Embedding hatten wir eine Tabelle mit 150.000 Zeilen, jede ein 4.096-Vektor, und haben für jedes Token die passende Zeile rausgeholt.

Beim LM-Head wird der finale Vektor mit jeder Zeile dieser Tabelle verglichen. Für jede Zeile fällt ein Score raus, wie gut der Vektor zu diesem Token passt. Am Ende: 150.000 Scores. Einer pro Token im Vokabular.

lm_head = nn.Linear(4096, 150_000)
logits  = lm_head(final_vector)   # [150000]

Diese Scores heißen Logits. Können beliebig groß oder klein sein, positiv oder negativ.

Aus Scores werden Wahrscheinlichkeiten

Logits sind noch nicht brauchbar. Die müssen in Wahrscheinlichkeiten umgewandelt werden. Wieder Softmax, die Operation die wir schon bei Attention hatten. Nimmt 150.000 rohe Scores und macht 150.000 Werte draus die zusammen Eins ergeben.

probs = F.softmax(logits, dim=-1)

Für unseren Satz nach dem Punkt könnte das so aussehen:

" Sie"        0,11
"\n"           0,10
" Schön"      0,07
" Ich"        0,06
"<eos>"       0,06
" Aber"       0,04
" Das"        0,04
...

Plus 150.000 weitere Tokens, alle mit winzigen Wahrscheinlichkeiten.

Kein Token liegt weit vorn. Verteilung ist flach. Passt auch, nach einem fertigen Satz mit Punkt gibt es halt viele mögliche Fortsetzungen. Neuer Satz, Vorschlag, Ende des Texts. Das Modell ist sich unsicher und die Verteilung zeigt das.

Einfach das wahrscheinlichste nehmen?

Könnte man machen. Heißt Greedy Decoding. Funktioniert, aber die Texte werden schnell langweilig. Das Modell findet einen sicheren Pfad und läuft den immer wieder. Loops, Wiederholungen, da ist skein Leben drin.

Stattdessen wird gewürfelt. Wenn „ Sie" 11 Prozent hat, wird in 11 Prozent der Fälle „ Sie" gezogen. 10 Prozent für Zeilenumbruch, wird in 10 Prozent gezogen. Und so weiter.

next_token = torch.multinomial(probs, num_samples=1)

multinomial macht genau das. Du gibst Wahrscheinlichkeiten rein und kriegst einen zufällig gezogenen Index zurück, gewichtet nach diesen Wahrscheinlichkeiten. Das nennt sich Sampling.

Temperatur

Der bekannteste Regler. Eine Zahl die die Logits vor der Softmax teilt. Klingt nach nix, ändert aber alles.

probs = F.softmax(logits / temperature, dim=-1)

Temperatur 1: Verteilung bleibt wie sie ist.

Temperatur runter, zB 0,3: Unterschiede werden krasser. Was schon wahrscheinlich war wird noch wahrscheinlicher. Aus 11 Prozent für „ Sie" werden vielleicht 35. Bei Temperatur 0 ist es reines Greedy, wahrscheinlichstes Token kriegt 100 Prozent.

Temperatur hoch, zB 1,5: Verteilung wird flacher. Aus 11 und 4 Prozent werden vielleicht 8 und 6. Sachen die vorher keine Chance hatten kriegen plötzlich eine.

Niedrige Temperatur = konservativ und vorhersagbar. Hohe Temperatur = experimentell und überraschend. Für Fakten-Aufgaben niedrig, für kreatives Schreiben hoch. Fertig.

Top-k und Top-p

Problem bei hoher Temperatur: auch totaler Unsinn kann in die Auswahl rutschen. „ Schraubendreher" hat ne winzige Wahrscheinlichkeit, nicht null, und bei genug Würfen taucht es irgendwann auf.

Deshalb gibts zwei Regler die das Kandidatenfeld einschränken bevor gewürfelt wird.

Top-k ist hart. Bei k=50 kommen nur die Top 50 Token in Frage. Rest wird auf null gesetzt.

Top-p ist weicher. Bei p=0,9 nimmst du so viele der wahrscheinlichsten Token bis ihre Wahrscheinlichkeiten zusammen 90 Prozent ergeben. Kann 5 Token sein wenn die Verteilung scharf ist oder 200 wenn sie flach ist. Passt sich automatisch an.

Top-p ist heute Standard. Übliche Kombi: Temperatur 0,7, Top-p 0,9.

Dann gibts noch Repetition Penalty und so Zeug das verhindert dass das Modell sich wiederholt. Wen juckt das im Detail, wichtig ist dass es existiert.

Der Loop

Wir haben so getan als würde der Satz einmal durchs Modell gehen und am Ende fällt ein Wort raus. Stimmt nicht ganz.

Wenn das Modell „ Sie" gewählt hat ist es nicht fertig. Es soll ja weiter schreiben. Also wird „ Sie" hinten angehängt, aus „Python ist eine schöne Sprache." wird „Python ist eine schöne Sprache. Sie", und die ganze Reise geht von vorne los. Das neue Token muss durch alle 80 Blöcke. Am Ende wieder eine Verteilung, wieder würfeln. Vielleicht „ ist". Dann „ einfach". Und so weiter.

while not done:
    logits      = model(input_ids)[:, -1, :]
    probs       = F.softmax(logits / temperature, dim=-1)
    next_token  = torch.multinomial(probs, num_samples=1)
    input_ids   = torch.cat([input_ids, next_token], dim=-1)
    done        = (next_token.item() == eos_id)

Das heißt autoregressive Generierung. Jeder Schritt ein Token. Der ganze Apparat aus 80 Blöcken läuft einmal pro Token. Ein Absatz bedeutet hunderte komplette Durchläufe.

Deshalb sind LLMs so rechenintensiv. Nicht weil ein Durchlauf teuer wäre, sondern weil jeder Satz aus vielen Durchläufen besteht.

Und deshalb wird in Token abgerechnet. Ein Token raus = ein Durchlauf durchs ganze Modell. Token sind nicht nur was rein und raus geht. Token sind das was bezahlt wird.

Und das wars

„Large Language Model" ist eigentlich ein irreführender Name. Ehrlicher wäre meiner Meinung nach „Large Token Model".

Was das Modell verarbeitet sind keine Wörter und keine Sprache. Es sind Token die zu Vektoren werden, die sich gegenseitig betrachten, die durch dutzende Layer geschoben werden, die am Ende eine Verteilung über mögliche nächste Token erzeugen aus der eins gewürfelt wird. Dann von vorne.

Sprache ist die Form in der wir mit dem Modell reden. Token sind die Form in der das Modell rechnet.

Wenn du jetzt in PyTorch ein nn.Embedding, nn.Linear oder F.softmax siehst weißt du was es tut und wo es in der Reise steht. Und dann kannst du anfangen dir den ganzen Kram selber zusammenzubauen.

Aus acht Token wurde ein neuntes. Dann ein zehntes. Dann ein elftes.
